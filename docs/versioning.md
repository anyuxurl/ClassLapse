# ClassLapse 版本管理

## 版本来源

`src/ClassLapse/ClassLapse.csproj` 是版本号的唯一事实来源：

```xml
<Version>0.4.0</Version>
<AssemblyVersion>0.4.0.0</AssemblyVersion>
<FileVersion>0.4.0.0</FileVersion>
```

- 设置窗口从程序集版本显示当前版本。
- `publish.ps1` 从 `<Version>` 读取压缩包文件名。
- 发布脚本读不到版本时必须失败，不能使用硬编码的旧版本兜底。
- `app.manifest` 的四段版本与程序集版本同步。

## 编号规则

正式版本采用 `主版本.次版本.修订号`：

- `0.x.0`：尚未进入 1.0 阶段的重要功能或产品里程碑，例如计划系统重构、视频合成、长期运行可靠性。
- `0.x.y`：不改变产品定位的小型修复，例如发布脚本、界面或特定异常处理修正。
- `1.0.0`：经过真实教室环境的长期连续运行，关键恢复、备份和部署流程均完成验证。

Git 标签统一使用 `v` 前缀，例如 `v0.4.0`；代码和文件属性使用 `0.4.0`。

## 发布流程

1. 确认计划纳入本次发布的功能和修复已经完成。
2. 更新 `ClassLapse.csproj`、`app.manifest` 和 `CHANGELOG.md`。
3. 运行 `dotnet test -c Release`。
4. 运行 `./publish.ps1 -Zip`，确认 EXE 文件属性和压缩包名称中的版本一致。
5. 提交版本变更，推荐提交信息：`chore(release): prepare vX.Y.Z`。
6. 在该提交上创建带说明的标签：`git tag -a vX.Y.Z -m "ClassLapse vX.Y.Z"`。
7. 推送提交和标签：`git push origin main`、`git push origin vX.Y.Z`。
8. 记录发布产物大小、SHA256 和实机验收结果。

标签必须指向已经包含正确版本元数据与更新日志的提交，不能先打标签再修改版本文件。
