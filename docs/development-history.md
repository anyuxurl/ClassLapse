# ClassLapse 开发历程

## 项目的出发点

ClassLapse 最初是一个运行在希沃白板上的静默定时拍照工具。随着目标逐渐明确，它不再只是一个普通的采集程序，而是要在整个高三学年持续记录教室，在毕业时把素材合成为一段属于班级的延时视频。

这个目标改变了项目的评价标准：功能数量不是最重要的，最重要的是一年后素材仍然完整、机位和时间连续、异常可以恢复，并且最终确实能够整理成有意义的视频。

## 2026-05-17：从零到首个可部署版本

开发在一天内按 M0 至 M7 完成了第一条完整链路：

- **M0 · 工程骨架**：建立 .NET 8 WPF 应用、测试工程和基础依赖。
- **M1 · 摄像头能力**：枚举摄像头、异步拍照，并提供开发命令行。
- **M2 · 调度与配置**：把是否应该拍照设计为纯决策逻辑，加入原子 JSON 配置和定时节拍。
- **M3 · 托盘应用**：连接调度器与摄像头，形成静默运行的桌面程序。
- **M4 · 设置界面**：加入计划、摄像头、存储和关于页面，以及校验和测试拍照。
- **M5 · 首次使用体验**：加入首次运行向导、多时段计划和设备最高分辨率拍摄。
- **M6 · 运维能力**：加入用户级开机自启、存储清理和按日滚动日志。
- **M7 · 发布完善**：补齐 README、希沃部署指南和单文件发布脚本。

提交 `18fea93` 被标记为 `v0.1.0`，这是第一个能够实际部署到教室设备的版本。

## 2026-05-30：从固定作息到真实学校计划

原有的全局拍摄时段难以表达早自习、上午、下午、晚自习、打铃和临时重叠计划。M8 将计划系统重构为独立条目：每个条目拥有自己的模式、星期、时段或时间点，并允许相互重叠。

提交 `10ce340` 构成逻辑版本 `0.2.0`。它让 ClassLapse 从“按几个固定窗口拍照”升级为能够描述真实学校作息的计划系统。

## 2026-05-30 至 2026-07-14：完成延时视频闭环

M9 在程序内加入多日延时视频合成。用户可以选择若干天甚至整个学期的照片，通过 FFmpeg 在后台生成 MP4，并查看进度或取消任务，正常拍照不会因此中断。

随后加入实时拍摄时间戳水印。项目曾短暂包含“给已有图片批量补水印”的一次性迁移工具；旧照片处理完成后，这套功能被删除，只保留长期需要的实时水印，避免增加无意义的维护面。

这些能力在合并提交 `ea14023` 汇合到 `main`，构成逻辑版本 `0.3.0`：采集、标记和合成第一次形成完整闭环。

## 2026-07-14：从可用工具到一年时间胶囊

项目审阅发现，秒级文件名可能覆盖素材、摄像头释放可能无限等待、拍照失败缺少独立流水，程序异常退出后也不能自动恢复。对于普通工具，这些只是缺陷；对于只能记录一次的高三学年，它们意味着不可补拍的数据丢失。

因此长期运行可靠性成为最高优先级，并完成了以下改造：

- 照片使用毫秒级名称，冲突时追加序号，永不覆盖已有文件。
- 先写临时文件并刷新磁盘，再以原子方式提交正式照片。
- 手动和计划拍照共用串行通道，避免争抢摄像头。
- 每次成功和失败都写入独立拍摄流水。
- 托盘显示最后成功时间，启动后从素材恢复今日计数。
- 摄像头释放设置时间上限，防止驱动卡住整个调度器。
- 增加异常退出守护重启、正常退出标记、单实例和重启循环保护。
- Release 测试基线提升到 `73/73`。

提交 `ae73843` 完成了核心实现。包含正式版本元数据和开发记录的后续发布提交将成为 `v0.4.0` 标签目标，版本定位为“**一年运行可靠性版**”。

## 接下来的方向

后续开发继续围绕“一年后能否得到完整且有意义的视频”展开：

1. 每日素材健康报告、最长断档和磁盘余量预测。
2. 第二份备份、月度归档和文件完整性校验。
3. 按目标视频时长自动抽帧、坏帧过滤和亮度平滑。
4. 学校日历、节假日、考试周和临时停拍日期。
5. 为开学、运动会、成人礼和最后一节课等时刻添加事件标记。

当项目经过真实教室环境的长期连续运行，并验证故障恢复和备份流程后，再进入 `1.0.0`。
